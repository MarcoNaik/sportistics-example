# 0001 — MatchCallup uses the availability map as the only source of truth

`MatchCallup` originally carried both `playerIds: string[]` and `availability: Record<string, Availability>`. The two collections drifted — a player could be in `playerIds` with no availability entry, or set as `available` without being in `playerIds` — and every caller that asked "is this player in the callup?" had to know which collection was authoritative. We dropped `playerIds`: a player is in the callup iff they have an entry in `availability` (default state when added is `pending`). Locality wins over normalization here because the data is small (bounded by roster size) and all rules now live in `src/domain/callup.ts` — there is no second collection to keep aligned. A future suggestion to "normalize this with a separate playerIds array" should not be reopened without showing two collections actually solve a problem the single map can't.
