# 0002 — Callup persists as flat `(matchId, playerId)` tuples at the storage Adapter

ADR 0001 fixed the in-memory shape of `MatchCallup.availability` as a single `Record<playerId, Availability>` keyed by player; that decision still holds at the domain Interface, where the frontend reads and writes a per-match dict. But Struere is the storage Adapter for this app, and the agents that drive Sportistics — WhatsApp callup confirmation, voice substitute calls, the coach stats widget — operate on individual `(matchId, playerId)` tuples, not on whole callups. Their natural tools are `set_availability(matchId, playerId, value)`, `get_replacement_candidates(matchId)`, and server-side filters like "every match where Ramirez is `unavailable`." Persisting Callup as one entity per match with a `players[]` array inside would force every single-player change through a whole-callup read-modify-write — race-prone under concurrent agent activity and forcing each tool call to load the full callup just to flip one entry. We therefore persist `Callup` as one row per `(matchId, playerId)` tuple — `Callup { matchId, playerId, availability, notes? }` — so agents can update a single tuple atomically and Struere's `entity.query` does the filtering server-side. The Callup Module in `src/domain/callup.ts` is the only mediator between the two shapes: `fromRows` aggregates tuples into the `MatchCallup` dict on read, `toRows` splits a `MatchCallup` back into tuples on write, and the wrapper in `src/lib/struere.ts` exposes only per-tuple methods (`setAvailability`, `removeEntry`, `removeForMatch`, `removeForPlayer`). Single-source-of-truth still holds at the Interface — there's still exactly one logical availability per `(matchId, playerId)` pair, enforced at the Adapter by querying the existing row before insert (last-write-wins on the tuple is fine for this app's traffic; a server-side trigger would tighten in production). This is not a retraction of ADR 0001: ADR 0001 forbade two collections in the same in-memory record drifting against each other, whereas this ADR is one logical collection in two representations — `Record` in memory, flat rows on the wire, with the Module as the only mediator. Note: TrainingSession does not follow the same flattening. Its per-player loads stay as an array-of-records inside one entity per session because the access pattern is different — the coach reads a whole training's loads together (the Load UI shows the full session at once), there is no agent updating individual loads tuple-at-a-time, and the array fits Struere's closed `defineData` JSON Schema directly.
